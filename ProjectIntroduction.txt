## 一、系统架构设计

### 1. 多智能体协作架构（Agent Orchestra）

```mermaid
graph TD
    User[用户] --> Web[Web前端]
    Web --> API[FastAPI网关]
    
    subgraph "智能体协作层 LangGraph"
    API --> Coordinator[协调器Agent<br/>状态机管理]
    Coordinator --> Analyzer[需求分析Agent<br/>分析小说类型/风格]
    Coordinator --> Architect[剧情架构Agent<br/>世界观/主线/冲突]
    Coordinator --> Character[角色设计Agent<br/>人设/关系/成长弧]
    Coordinator --> Outline[大纲生成Agent<br/>分卷/章节节点]
    Coordinator --> Writer[章节撰写Agent<br/>正文生成]
    Coordinator --> Consistency[一致性检查Agent<br/>剧情连续性]
    Coordinator --> Memory[记忆管理Agent<br/>RAG检索/摘要]
    end
    
    subgraph "数据层"
    MySQL[(MySQL<br/>小说/章节/角色)]
    VectorDB[(向量数据库<br/>Qdrant/Pinecone<br/>剧情记忆)]
    Redis[(Redis<br/>会话状态)]
    end
    
    Writer --> MySQL
    Writer --> VectorDB
    Memory --> VectorDB
```

### 2. 关键设计模式

**长文本记忆解决方案**：
- **分层摘要**：每10章生成一次中级摘要，每100章生成高级摘要
- **向量检索**：关键剧情节点、角色状态向量化存储
- **记忆窗口**：智能体始终携带"近期10章 + 关键剧情节点 + 角色当前状态"

**状态机工作流**：
```python
# 状态定义
STATES = [
    "AWAITING_GENRE",      # 等待确认类型
    "DISCUSSING_PLOT",     # 商讨剧情
    "DESIGNING_CHARS",     # 设计角色
    "GENERATING_OUTLINE",  # 生成大纲
    "WRITING_CHAPTER",     # 撰写章节
    "REVIEWING",           # 人工审核
    "CONTINUING"           # 自动续写
]
```

## 二、数据库设计（MySQL Schema）

```sql
-- 小说项目表
CREATE TABLE novels (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    title VARCHAR(255),
    genre VARCHAR(100),          -- 玄幻/科幻/都市等
    style_prompt TEXT,           -- 风格提示词
    total_chapters INT DEFAULT 0,
    current_chapter INT DEFAULT 0,
    status ENUM('planning', 'writing', 'paused', 'completed'),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 章节表（分表策略：每本小说独立表或按百万章分表）
CREATE TABLE chapters (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    novel_id BIGINT,
    chapter_number INT,
    title VARCHAR(255),
    content LONGTEXT,            -- 章节正文
    summary TEXT,                -- 本章摘要
    key_events JSON,             -- 关键事件
    characters_present JSON,     -- 出场角色
    word_count INT,
    status ENUM('draft', 'review', 'published'),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_novel_chapter (novel_id, chapter_number)
);

-- 角色表
CREATE TABLE characters (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    novel_id BIGINT,
    name VARCHAR(100),
    role_type ENUM('protagonist', 'antagonist', 'supporting', 'minor'),
    profile JSON,                -- 详细人设
    current_status TEXT,         -- 当前状态（动态更新）
    first_appearance INT,        -- 首次出场章节
    arc_progress FLOAT           -- 角色弧光进度 0-1
);

-- 剧情大纲表
CREATE TABLE outlines (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    novel_id BIGINT,
    volume_number INT,           -- 第几卷
    volume_title VARCHAR(255),
    arcs JSON,                   -- 剧情弧数组
    key_points TEXT,             -- 关键节点
    target_chapters INT          -- 预计章节数
);

-- 记忆向量表（与向量数据库同步）
CREATE TABLE memory_nodes (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    novel_id BIGINT,
    node_type ENUM('plot_point', 'character_moment', 'world_building', 'mystery'),
    content TEXT,
    chapter_range VARCHAR(50),   -- 涉及章节范围
    importance_score FLOAT,      -- 重要性 0-1
    embedding_id VARCHAR(255)    -- 向量库ID
);
```

## 五、关键注意事项

1. **校验测试**：每次功能模块完善后，要进行测试，确定内容是否完善。
2. **数据库性能**：1000章×5000字=500万字，建议：
   - MySQL使用TEXT类型存储内容
   - 为(novel_id, chapter_number)添加复合索引
   - 考虑按novel_id分表或归档策略

3. **一致性保障**：
   - 在Consistency Agent中实现"设定锁"：核心设定（如修为体系）变更需人工确认
   - 定期进行"剧情审计"：检索所有包含某角色的章节检查行为一致性

4. **计划确认**：你需要在做的过程中不断回顾本计划，并在Progress.txt中记录你的进度
